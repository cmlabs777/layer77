Pain 1 · SNS 모니터링
수동 모니터링의 한계
  • 블로그·인스타·트위터 등 채널마다 직접 접속해 검색·확인

  • 모니터링 깊이와 빈도가 담당자 여건에 따라 편차 발생

브랜드 뉴스 사각지대
  • 브랜드 관련 뉴스를 수동으로 검색해야 하는 구조
  • 팀 전체에 정기적으로 공유할 수 있는 프로세스 부재
  • 중요 보도 누락 시 대응 타이밍 놓침
  • 담당자 부재 시 모니터링 공백 발생

01 · What We Built
실제로 구축·운영 중인 시스템
System 01 · 계획 수행
SNS 브랜드 모니터링
자동화 시스템
Make.com + Apify로 6개 채널 게시글을 매일 자동 수집. Claude API가 브랜드 관련성 필터링 → 감성 분류(긍정/중립/부정) → 트렌드 인사이트 생성 → JSON 리포트 출력. Teams Adaptive Card로 매일 오전 자동 발송.
Make.com Apify Claude API Naver API Google API Teams Webhook
System 02 · 계획 수행 BONUS
브랜드 뉴스 클리핑
자동화 시스템
네이버 뉴스 API로 브랜드 관련 기사를 매일 오전 9시 자동 수집해 Teams로 발송. Claude를 활용한 시스템 구성, Google Apps Script 기반으로 별도 서버 없이 운영.
Naver News API Google Apps Script Claude Teams Webhook

04 · Learning & Iteration
시도 → 실패 → 개선 과정
1
문제
LLM JSON 출력 불안정
Claude API가 백틱(```) 코드블록을 포함하거나 JSON 구문 오류를 발생시켜 자동화 파이프라인이 중단되는 상황이 반복됐습니다.
개선
프롬프트 엔지니어링으로 안정화
출력 형식을 명시하는 프롬프트를 반복 개선해 오류 빈도를 낮췄습니다. 근본적 해결을 위해 Tool Use(Function Calling) 도입을 2차 목표로 설정했습니다.
2
문제
AI 카운팅 오류
LLM에게 게시글 수 집계를 맡기면 실제 데이터와 다른 숫자를 출력하는 경우가 발생했습니다.
개선
역할 분리 — 집계는 코드, 분석은 AI
집계·카운팅은 Make.com 워크플로우에서 처리하고, AI에게는 언어 이해와 인사이트 생성만 위임하는 방식으로 전환해 정확도를 높였습니다.
3
문제
반복 테스트로 인한 비용 누적
코드 한 줄 수정 후 전체 파이프라인을 재실행해야 하는 구조 때문에 Apify·Claude API 호출이 반복되어 테스트 비용이 예상보다 높았습니다.
개선
샘플 데이터 고정 + 단위 테스트 분리
테스트용 샘플 데이터를 고정해 수집 단계를 건너뛰는 방식을 도입하고, 분석 모듈을 독립적으로 테스트할 수 있도록 구조를 개선했습니다.
🖥
05 · Control Center
종합 관제 대시보드
전용 대시보드를 통해 시스템 안정성을 상시 모니터링합니다
두 시스템의 실행 상태 및 서비스 안정성을 하나의 관제 화면에서 통합 확인하고, 이상 발생 시 즉시 수동 개입이 가능한 구조로 운영합니다. 시스템이 추가되는 경우, 대시보드에 포함되어 함께 관제됩니다.
