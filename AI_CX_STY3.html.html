Pain 1 · SNS 모니터링
수동 모니터링의 한계
  • 블로그·인스타·트위터 등 채널마다 직접 접속해 검색·확인
  • 담당자 1인이 주 20시간 이상 단순 반복 작업에 소요
  • 모니터링 깊이와 빈도가 담당자 여건에 따라 편차 발생
  • 이슈 발생 시 인지 시점이 늦어 초기 대응 불가
Pain 2 · 뉴스 클리핑
브랜드 뉴스 사각지대
  • 브랜드 관련 뉴스를 수동으로 검색해야 하는 구조
  • 팀 전체에 정기 공유할 수 있는 프로세스 부재
  • 중요 보도 누락 시 대응 타이밍 놓침
  • 담당자 부재 시 모니터링 공백 발생
핵심은 "사람이 반복하던 일을 시스템으로 대체"하는 것이 아니라, 기존에는 불가능했던 수준의 분석·인사이트를 AI로 실현하는 것이었습니다.

What We Built

실제로 구축·운영 중인
두 가지 시스템

01
계획 수행
SNS 브랜드 모니터링 자동화 시스템
Make.com + Apify로 6개 채널 게시글을 매일 자동 수집. Claude API가 브랜드 관련성 필터링 → 감성 분류(긍정/중립/부정) → 트렌드 인사이트 생성 → JSON 리포트 출력. Teams Adaptive Card로 매일 오전 자동 발송.
Make.com Apify Claude API Naver API Google API Teams Webhook
02
계획 外 추가
브랜드 뉴스 클리핑 자동화 시스템
당초 계획에 없던 기능으로, 진행 중 팀 니즈를 확인하고 추가 개발. 네이버 뉴스 API로 브랜드 관련 기사를 매일 오전 9시 자동 수집해 Teams로 발송. Claude를 활용한 시스템 구성, Google Apps Script 기반으로 별도 서버 없이 운영.
Naver News API Google Apps Script Claude Teams Webhook

매일 자동
01
1단계 필터링 — 수집된 전체 게시글 중 브랜드와 실제 관련된 게시글만 선별. 유사 단어·무관 콘텐츠 제거.
02
감성 분류 — 각 게시글을 긍정/중립/부정으로 분류하고 플랫폼별 감성 비율 산출.
03
인사이트 생성 — 오늘의 핵심 트렌드, 브랜드 전략 시사점, 모니터링 필요 이슈를 자연어로 작성.
04
구조화 출력 — 분석 결과를 JSON 포맷으로 출력해 Teams Adaptive Card에 자동 맵핑.

Learning & Iteration

시도 → 실패 → 개선,
세 번의 반복

1
문제
LLM JSON 출력 불안정
Claude API가 백틱(```) 코드블록을 포함하거나 JSON 구문 오류를 발생시켜 자동화 파이프라인이 중단되는 상황이 반복됐습니다.
개선
프롬프트 엔지니어링으로 안정화
출력 형식을 명시하는 프롬프트를 반복 개선해 오류 빈도를 낮췄습니다. 근본적 해결을 위해 Tool Use 도입을 2차 목표로 설정했습니다.
2
문제
AI 카운팅 오류
LLM에게 게시글 수 집계를 맡기면 실제 데이터와 다른 숫자를 출력하는 경우가 발생했습니다.
개선
집계는 코드, 분석은 AI
집계·카운팅은 Make.com 워크플로우에서 처리하고, AI에게는 언어 이해와 인사이트 생성만 위임하는 방식으로 전환했습니다.
3
문제
반복 테스트로 인한 비용 누적
코드 한 줄 수정 후 전체 파이프라인을 재실행해야 해서 Apify·Claude API 호출이 반복되어 테스트 비용이 예상보다 높았습니다.
개선
샘플 데이터 고정 + 단위 테스트 분리
테스트용 샘플 데이터를 고정해 수집 단계를 건너뛰고, 분석 모듈을 독립적으로 테스트할 수 있도록 구조를 개선했습니다.
